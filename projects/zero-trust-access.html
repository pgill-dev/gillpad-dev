ZERO TRUST · REMOTE ACCESS · PLATFORM ENGINEERING

Zero Trust Remote Access Platform

Designed and operated an identity-first remote access platform using Cloudflare Access, Apache Guacamole, Docker, PostgreSQL, and segmented Proxmox infrastructure without exposing inbound management ports.

Cloudflare Access Cloudflare Tunnel Apache Guacamole Docker PostgreSQL Proxmox VE OPNsense Linux

THE PROBLEM

Secure access without exposing the lab.

The environment needed browser-based SSH and RDP access for multiple users while keeping the Proxmox management plane and internal systems off the public Internet.

Traditional port forwarding would have created direct exposure for management protocols. A mesh VPN would have reduced that exposure, but still introduced endpoint-agent requirements and broader network trust. The selected design instead enforced identity before application access and used a single controlled ingress path.

Three authentication layers

Users pass Cloudflare identity checks, Guacamole authentication, and native endpoint authentication before reaching a target system.

No exposed management services

SSH, RDP, and virtualization management interfaces remain internal. Cloudflare Tunnel maintains an outbound connection to the edge.

SECURITY CONTROLS

Designed to limit trust and blast radius.

Identity-first access

MFA and explicit allow rules are evaluated before any application traffic reaches the lab.

Default-deny policy

Users must be intentionally authorized. A final deny policy prevents accidental access outside approved identities.

Individual accounts

Shared credentials were avoided. Guacamole users received scoped access to only the connections required for their role.

Central revocation

Access can be removed at the Cloudflare edge, then disabled inside Guacamole, providing a fast and layered revocation path.

Segmented management path

The bastion acts as the access choke point between authenticated users and internal systems.

Persistent authorization data

PostgreSQL stores Guacamole identities, connections, and permissions across container restarts and host reboots.

ENGINEERING CHALLENGES

Most failures crossed more than one layer.

  • Corrected DNS records associated with an incorrect tunnel identifier.
  • Resolved routing failures caused by Cloudflare forwarding to an unreachable backend after container network changes.
  • Accounted for Guacamole being served from a specific application context path rather than the root hostname.
  • Verified PostgreSQL persistence instead of assuming authentication failures meant the database had been lost.
  • Reordered Access policies so explicit allow rules were evaluated before the final deny rule.
  • Diagnosed silent Layer 2 packet loss caused by hypervisor-level firewall filtering on an isolated bridge.
  • Stabilized long-running access sessions through server-side XRDP tuning and persistent SSH workflows using tmux.
Key troubleshooting principle: validate each layer independently—container runtime, application, database, network path, identity policy, and endpoint authentication.

VALIDATION

Proved the design through controlled tests.

Access enforcement

Tested unauthenticated and authenticated browser sessions to confirm Cloudflare Access could not be bypassed.

Protocol access

Validated SSH and RDP through Guacamole while keeping target services unavailable directly from the Internet.

Reboot recovery

Confirmed container startup, database persistence, tunnel routing, and application availability after restart.

User lifecycle

Tested creation, scoped authorization, and centralized revocation of individual user access.

Network segmentation

Verified internal bridge operation, bastion reachability, and controlled management access through firewall logs and connection tests.

Operational resilience

Documented recovery procedures, troubleshooting steps, and a stable baseline suitable for snapshots and future expansion.

OUTCOME

A repeatable, controlled ingress platform.

The completed platform provides browser-based, multi-user access to internal lab systems with MFA, layered authentication, scoped permissions, centralized revocation, and no inbound management ports.

The design also established a reusable operating model for future training environments, malware-analysis workflows, and isolated security ranges.

LESSONS LEARNED

What this project reinforced.

  • Identity, application, and endpoint authentication should remain separate.
  • Policy order must be explicit and validated.
  • ARP failure should be treated as a Layer 2 problem before routing is blamed.
  • Container health does not guarantee application reachability.
  • Persistent data and persistent network addressing are different concerns.
  • CLI state is often more authoritative than a stale management interface.
  • Recovery procedures are part of the architecture, not an afterthought.
