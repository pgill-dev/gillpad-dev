LAB ARCHITECTURE OVERVIEW

Layered Home Lab Architecture

A high-level view of the Gillpad infrastructure lab, showing how the public portfolio, secure remote access path, Proxmox virtualization layer, network segmentation, lab workloads, and recovery model fit together without exposing private IP addresses or sensitive internal details.

Status: Operational Type: Architecture Overview Role: Architect / Operator Focus: Segmentation and Recovery

HIGH-LEVEL DESIGN

The lab is organized as layered infrastructure, not one flat network.

The environment separates public presentation, secure remote access, core infrastructure, network segments, lab workloads, and recovery workflows. This keeps the public website separate from private lab services while still allowing the lab to be used for engineering, testing, documentation, and student-friendly demonstrations.

HOW TO READ THE DIAGRAM

Each layer has a specific job.

Public Presentation Layer

The public website is hosted through Cloudflare Pages. It showcases the lab and case studies without depending on direct access to the private home lab.

Secure Remote Access Layer

Private access flows through Cloudflare Access, Cloudflare Tunnel, Apache Guacamole, and a bastion host. This creates a controlled path into the environment without exposing management services directly.

Core Infrastructure Layer

Proxmox VE provides the virtualization platform, OPNsense handles routing and segmentation, PBS supports recovery, and the management VM provides an operator workspace.

Virtual Network Segments

Workloads are separated by function. Management, internal services, and security ranges do not all sit together on one flat network.

Lab Workload Layer

The lab contains purpose-built workloads, including Kubernetes, GOAD, malware sandboxing, general VMs, and internal Proxmox tooling.

Operations / Recovery Layer

Backups, verification, snapshots, rebuilds, and runbooks are treated as part of the architecture instead of afterthoughts.

SECURE ACCESS PATH

Private lab access is brokered through a controlled ingress path.

The access model separates the public portfolio from private lab administration. Visitors can view the website publicly, but private lab access requires the secure access path. 

flowchart LR
  USER[Operator] --> CFA[Cloudflare Access]
  CFA --> TUN[Cloudflare Tunnel]
  TUN --> GUAC[Apache Guacamole]
  GUAC --> BAST[Bastion / Jump Box]
  BAST --> PVE[Proxmox / Internal Systems]

Identity first

Access is controlled before a user reaches the internal application path. The design favors identity-aware access over directly exposed administrative services.

No direct management exposure

SSH, RDP, and virtualization administration stay behind the secure path. The public website does not provide direct access to private infrastructure.

PROXMOX VM ARCHITECTURE

Proxmox acts as the platform that hosts the lab workloads.

The virtual machine architecture is organized by workload purpose. Each project has a role in the overall lab: secure access, automation, security testing, cloud-native learning, Windows/AD practice, or recovery validation.

Management workloads

Management and automation systems provide a controlled place to run tools, edit configuration, document changes, and operate the lab.

Security workloads

GOAD and sandbox systems are treated as security range workloads. They are designed for controlled testing and should not be trusted like normal infrastructure.

Cloud-native workloads

Kubernetes, Istio, Traefik, Helm, and related platform components support cloud-native traffic, routing, ingress, and service mesh learning.

Recovery workloads

PBS, snapshots, verification, and runbooks provide the operational safety net for rebuilding and recovering lab systems.

SEGMENTATION MODEL

The lab is intentionally separated by trust and purpose.

The environment uses segmentation to keep management systems, internal services, and intentionally vulnerable security ranges from collapsing into one flat space. This makes the lab easier to reason about, easier to troubleshoot, and safer to use for offensive and defensive testing.

Management Segment

Used for administration, automation, tooling, and controlled access to infrastructure services.

Internal Services Segment

Used for general lab services, cloud-native workloads, Linux/Windows systems, and supporting infrastructure.

Security Range Segment

Used for intentionally vulnerable or high-risk lab workloads such as GOAD, testing ranges, and attack-and-defense practice.

Recovery Boundary

Backups, snapshots, and rebuild workflows support experimentation without requiring every VM to be treated like permanent production.

STUDENT LEARNING PATH

The architecture is built to be explainable.

The lab is not just a collection of VMs. It is a teaching environment where each layer can be explained independently before showing how the layers interact.

Start with access

Students can first understand how users reach the environment and why direct exposure of management ports is avoided.

Then explain virtualization

Proxmox can be introduced as the platform that hosts isolated VMs, networks, testing ranges, and recovery workflows.

Then explain segmentation

Network segments show why management systems, service workloads, and vulnerable ranges should not all share the same trust boundary.

Then explain recovery

Snapshots, backups, rebuilds, and runbooks show how engineers make a lab safe to break and useful to learn from.

PROJECT SUMMARY

The lab architecture ties the portfolio together.

This architecture overview shows how the individual case studies fit into one larger engineering environment. The public website, secure access path, Proxmox platform, segmented networks, lab workloads, and recovery workflows all support the same goal: a safe, explainable, rebuildable lab for infrastructure engineering, cybersecurity practice, automation, and teaching.
