LAB SYSTEM MAP

Lab Architecture Overview

A layered overview of the Gillpad engineering lab showing how public presentation, secure remote access, firewalling, Proxmox virtualization, segmented networks, workload groups, and recovery operations fit together without exposing private infrastructure details.

Status: Operational Type: Architecture Overview Role: Architect / Operator Focus: Segmentation and Recoverability

PURPOSE

The lab is designed as a layered engineering environment, not a flat pile of VMs.

The environment supports infrastructure engineering, security testing, cloud-native experiments, automation, vulnerability management, and student-style demonstrations. The architecture separates the public portfolio from private lab access, uses controlled ingress paths for administration, and keeps higher-risk workloads inside dedicated network segments.

This page is intended to give a high-level map of how the lab is laid out without publishing sensitive details such as IP addresses, internal hostnames, credentials, or provider-specific private configuration.

HOW TO READ THE DIAGRAM

Each layer has a specific job.

Public / External Layer

The public website is hosted through Cloudflare Pages and is separate from the private home lab. It showcases the work without requiring inbound access to private infrastructure.

Secure Access Layer

Private lab access flows through Cloudflare Access, Cloudflare Tunnel, Apache Guacamole, and a bastion or jump box. This creates a controlled path into the environment instead of exposing management services directly.

Core Infrastructure Layer

Proxmox VE provides the virtualization platform, OPNsense handles firewalling and segmentation, PBS supports backup and verification, and the management VM supports automation and operator workflows.

Virtual Network Segments

Workloads are separated by function and risk. Management systems, normal internal services, and intentionally vulnerable security ranges are not treated as the same trust zone.

Workload / VM Layer

Workloads include the Proxmox management platform, Kubernetes and Istio testing, GOAD Active Directory range work, malware or sandbox experiments, and general Linux or Windows systems.

Operations / Recovery Layer

The lab is operated with backups, verification, snapshots, rebuild workflows, documentation, and runbooks so failures can be contained and recovered from.

ACCESS MODEL

Users do not directly land on internal management services. 

flowchart LR
  USER[Operator] --> ACCESS[Cloudflare Access]
  ACCESS --> TUN[Cloudflare Tunnel]
  TUN --> GUAC[Apache Guacamole]
  GUAC --> BASTION[Bastion / Jump Box]
  BASTION --> MGMT[Management VM]
  BASTION --> PVE[Proxmox VE]
  BASTION --> LAB[Internal Lab Systems]

The access model is designed around controlled entry. The public site is available to anyone, but private administration requires passing through the access stack before reaching internal systems. This keeps SSH, RDP, hypervisor management, and lab workloads away from direct public exposure.

SEGMENTATION MODEL

Different workloads live in different trust zones.

Management Segment

Contains systems used to administer, automate, or observe the lab. This segment is treated as higher trust and should not be mixed with intentionally vulnerable workloads.

Internal Services Segment

Hosts normal lab services, Linux and Windows systems, Kubernetes testing, and other general workloads that support engineering practice.

Security Range Segment

Contains intentionally vulnerable or adversary-emulation workloads such as GOAD. This area is designed to be breakable, rebuildable, and separated from the rest of the lab.

Sandbox / Malware Segment

Used for controlled experiments, analysis workflows, and isolated systems that should not be trusted by default.

Organizing workloads by purpose makes the lab easier to explain, troubleshoot, rebuild, and expand. It also helps students understand that infrastructure design is not just about creating VMs; it is about placing systems in the right context.

RECOVERY MODEL

The lab assumes things will break and plans for recovery.

Snapshots

Used before risky changes, testing, lab exercises, and intentionally destructive workflows so systems can be reverted quickly.

Backups

Proxmox Backup Server provides a recovery layer for protected workloads after they are managed under the Proxmox environment.

Rebuilds

Lab ranges are designed with rebuildability in mind. Security ranges and sandbox systems should be treated as disposable when necessary.

Runbooks

Documentation captures the intended access paths, validation steps, recovery checks, and operational lessons learned from each build.

STUDENT LEARNING PATH

The lab can be explained from the outside in.

Start with access

Students can first understand how a user safely reaches the lab through identity-aware access, tunneling, Guacamole, and a bastion host.

Move into virtualization

From there, students can see how Proxmox provides compute, networking, snapshots, templates, and VM lifecycle management.

Then learn segmentation

The network model shows why management systems, normal workloads, and intentionally vulnerable security ranges should be separated.

Finish with operations

Recovery, documentation, monitoring, validation, and rebuildability show that engineering work continues after the first successful deployment.
